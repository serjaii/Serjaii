From Monolith to REST API: Architecture & DevOps for a React Native Football App

In-depth analysis of migrating a monolithic Django system to a REST API architecture for React Native. Covers Jenkins CI/CD, Push Notifications, and Service Layers.

20 Apr 2026
4 minute read

Software development is never a static process; it evolves as user needs change. Recently, I undertook one of the most interesting architectural challenges in my main project (an advanced football league management platform): making the leap from a traditional monolithic Django web application to a mobile ecosystem with React Native.

This migration, developed under the feature/react-native-xports-software branch, was far from a simple aesthetic update. It required a deep refactoring, implementing a scalable REST API, isolating business logic into a dedicated Service Layer, developing complex algorithms, and integrating a continuous deployment (CI/CD) pipeline using Jenkins.

Here is a breakdown of the most relevant technical aspects of this evolution.

1. The Challenge: Decoupling a Monolith

Initially, the project operated on the classic Django MTV (Model-Template-View) pattern. All HTML was rendered server-side, and views contained both HTTP response logic and heavy database queries.

For a React Native mobile app to consume data efficiently, a REST API was strictly required. However, there was a catch: I couldn’t break the existing web functionality. The solution was adopting the Service Layer Design Pattern.

Separation of Concerns and Django REST Framework (DRF)

Instead of relying on massive views (the original views.py had nearly a thousand lines of code), I divided the logic.

  1. Modular Models: The monolithic models.py file was dissected into a package of cohesive modules (alineaciones.py, competicion.py, partidos.py, etc.).
  2. Service Layer: All heavy lifting was extracted into files like home_service.py and equipo_service.py. This layer is agnostic; it doesn’t care if it’s called from a web view, a CLI command, or an API endpoint.
  3. API Controllers: Using Django REST Framework, I created torneo/api/views.py. These controllers (ViewSets) are extremely lightweight; their only mission is to receive the HTTP request, call the Service Layer, and return JSON via Serializers.

2. Complex Algorithmic Features

A sports app isn’t just a standard “CRUD” (Create, Read, Update, Delete) application; it requires deep data analysis and algorithms.

Form and Streak Algorithm (racha_service.py)

To show which players are “on fire”, I implemented a service that algorithmically evaluates recent performance. The algorithm analyzes the last 5 matches played, granting Scores for specific milestones:

  • Hat-tricks and Braces: Multipliers for recent goals.
  • Consistency: Bonuses for scoring streaks across consecutive matches.
  • Team Performance: Players are rewarded if their team maintains winning or undefeated streaks. This calculation translates into a dynamic Rating (0-99) and automated descriptive tags (e.g., “Recent Hat-trick”, “3 matches undefeated”), delivering gaming-style metrics to the app in real-time.

Animated Draw Generator (Champions League Style)

In the admin panel, generating a boring calendar isn’t enough. Using Vanilla JavaScript, I developed generador_animado.js, which visually simulates a live tournament draw. Leveraging an event queue (colaAnimacion), promises, and DOM manipulation via AJAX, the script draws virtual “balls”, pairing teams or assigning them to groups with smooth visual transitions, before making the final database commit to Django to generate the schedule (calendario_service.py).

3. User Retention: Push Notifications

A modern mobile app needs to be proactive. To achieve this, we integrated the OneSignal REST API. From Django, using the notifications.py module, the server automatically dispatches JSON payloads to OneSignal’s servers.

This allows us to:

  • Global Notifications: Alert all registered users when a new tournament kicks off.
  • Tag-based Segmentation: If there is a schedule change or match results are published, the backend can send the alert exclusively to the devices of players belonging to the involved teams by using specific filters in the HTTP request.

4. DevOps and CI/CD: Automated Deployment with Jenkins

A project of this scale cannot rely on manual deployments by copying files over FTP or running SSH commands by hand. This is where the DevOps mindset comes in.

To achieve a “Zero Friction” philosophy, the repository includes a declarative Jenkinsfile that orchestrates the entire deployment to production automatically. When code reaches the main branch, Jenkins executes the following phases via a secure SSH agent:

  1. Repository Pull: Synchronizes the source code on the VPS.
  2. Dependency Management: Executes pip install -r requirements.txt to ensure all Python libraries are up to date.
  3. Persistence & Statics: Automatically runs manage.py migrate (to safely update the database schema) and collectstatic.
  4. Service Restart: Finally, it interacts with systemctl at the OS level (Linux) to restart the uwsgi or gunicorn service, bringing up the new version with zero downtime.

Conclusion

Migrating from a classic Django application to an API-First architecture tailored for React Native proves that a solid architectural foundation is vital. The extensive use of the Service Layer pattern, efficient backend algorithms, real-time push communications, and CI/CD automation with Jenkins have transformed a university project into a Full-Stack ecosystem worthy of any production environment.
