De Monolito a API REST: Arquitectura y DevOps para una App de Fútbol en React Native

Análisis profundo de la migración de un sistema monolítico en Django a una arquitectura de API REST para React Native. Incluye CI/CD con Jenkins, Push Notifications y Service Layers.

20 abr 2026
Tiempo de lectura 4 minutos

El desarrollo de software no es un proceso estático; evoluciona a medida que lo hacen las necesidades de los usuarios. Recientemente, llevé a cabo uno de los desafíos arquitectónicos más interesantes en mi proyecto principal (un gestor avanzado de ligas de fútbol): dar el salto de una aplicación web tradicional monolítica en Django hacia un ecosistema móvil con React Native.

Esta migración, desarrollada bajo la rama feature/react-native-xports-software, no fue un simple cambio estético. Requirió una refactorización profunda, implementando una API REST escalable, aislando la lógica de negocio en una capa de servicios (Service Layer), desarrollando algoritmos complejos e integrando un pipeline de despliegue continuo (CI/CD) con Jenkins.

A continuación, desgrano los aspectos técnicos más relevantes de esta evolución.

1. El Desafío: Desacoplar un Monolito

Inicialmente, el proyecto funcionaba con el patrón clásico MTV (Model-Template-View) de Django. Todo el HTML se renderizaba en el servidor y las vistas contenían tanto la respuesta HTTP como las pesadas consultas a base de datos.

Para que una aplicación móvil en React Native pudiera consumir los datos eficientemente, necesitaba una API REST. Sin embargo, esto presentaba un problema: no podía romper la web existente. La solución fue adoptar el Patrón de Diseño “Service Layer” (Capa de Servicios).

Separación de Responsabilidades y Django REST Framework (DRF)

En lugar de tener vistas gigantes (views.py original tenía cerca de mil líneas), dividí el código.

  1. Modelos modulares: El archivo monolítico models.py fue diseccionado en un paquete de módulos coherentes (alineaciones.py, competicion.py, partidos.py, etc.).
  2. Capa de Servicios: Toda la lógica pesada fue extraída a archivos como home_service.py y equipo_service.py. Esta capa es agnóstica; no le importa si es llamada desde una web, un comando CLI o una API.
  3. API Controllers: Usando Django REST Framework, creé torneo/api/views.py. Estos controladores (ViewSets) son extremadamente ligeros; su única misión es recibir la petición HTTP, llamar a la Capa de Servicios y devolver un JSON a través de Serializers.

2. Funciones Algorítmicas Complejas

Una app deportiva no es solo “CRUD” (Crear, Leer, Actualizar, Borrar); requiere análisis de datos y algoritmos.

Algoritmo de Forma y Rachas (racha_service.py)

Para mostrar qué jugadores están “on fire”, implementé un servicio que evalúa algorítmicamente el rendimiento reciente. El algoritmo analiza los últimos 5 partidos jugados, otorgando puntos (Score) por hitos concretos:

  • Hat-tricks y Dobletes: Multiplicadores por goles recientes.
  • Constancia: Bonificaciones por rachas de partidos consecutivos marcando.
  • Rendimiento Colectivo: Se premia si el equipo mantiene rachas de victorias o imbatibilidad. Este cálculo se traduce en un Rating dinámico (0-99) y etiquetas descriptivas automatizadas (ej: “Hat-trick reciente”, “3 partidos invicto”), devolviendo a la app métricas de estilo gaming en tiempo real.

Generador Animado de Sorteos (Champions League Style)

En el panel de administración, no basta con generar un calendario aburrido. Usando Vanilla JavaScript, desarrollé generador_animado.js, que simula visualmente un sorteo en vivo. Haciendo uso de una cola de eventos (colaAnimacion), promesas y manipulación del DOM mediante AJAX, el script extrae “bolitas” virtuales, emparejando equipos o asignándolos a grupos con transiciones visuales, antes de impactar definitivamente a la base de datos de Django para generar el calendario (calendario_service.py).

3. Retención de Usuarios: Notificaciones Push

Una aplicación móvil necesita ser proactiva. Para ello, integramos la API REST de OneSignal. Desde Django, mediante el módulo notifications.py, el servidor despacha automáticamente payloads en JSON a los servidores de OneSignal.

Esto nos permite:

  • Notificaciones Globales: Avisar a todos los usuarios registrados cuando arranca un nuevo torneo.
  • Segmentación por Tags: Si hay un cambio de horario o se publica el acta de un partido, el backend puede enviar la alerta exclusivamente a los dispositivos de los jugadores que pertenecen a los equipos involucrados, usando filtros específicos en la petición HTTP.

4. DevOps y CI/CD: Despliegue Automatizado con Jenkins

Un proyecto de esta envergadura no se puede desplegar manualmente copiando archivos por FTP o entrando por SSH a lanzar comandos a mano. Aquí entra la mentalidad DevOps.

Para lograr una filosofía de “Cero Fricción”, el repositorio cuenta con un Jenkinsfile declarativo que orquesta todo el despliegue a producción de forma desatendida. Cuando el código llega a la rama main, Jenkins ejecuta las siguientes fases mediante un agente SSH seguro:

  1. Pull del Repositorio: Sincroniza el código fuente en el VPS.
  2. Gestión de Dependencias: Ejecuta pip install -r requirements.txt para asegurar que todas las librerías de Python están al día.
  3. Persistencia y Estáticos: Lanza de forma automática manage.py migrate (para actualizar la base de datos de forma segura) y collectstatic.
  4. Reinicio de Servicios: Por último, interactúa con systemctl a nivel de sistema (Linux) para reiniciar el servicio uwsgi o gunicorn, levantando la nueva versión sin caída de servicio.

Conclusión

Migrar de una aplicación clásica en Django hacia una arquitectura API-First diseñada para React Native demuestra que las buenas bases arquitectónicas son vitales. El uso extensivo del patrón de capa de servicios, algoritmos eficientes en backend, integración de comunicaciones en tiempo real (Push) y la automatización CI/CD con Jenkins han transformado un proyecto universitario en un ecosistema Full Stack digno de cualquier entorno de producción.
